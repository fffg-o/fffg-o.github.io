#### 我建议你的项目定位

项目一句话可以先定成：

**Bag of Fate / Packrun**  
一个“背包管理 + roguelike 卡牌战斗”游戏：玩家在局内收集卡牌、遗物、背包物品，通过背包摆放关系影响战斗牌组、费用、触发和成长；局外把关键成果写到 Solana，形成资产、赛季榜和可交易/可组合的游戏身份。

这个定位好处是：

1. **玩法本身就成立**，不会变成“为了上链而上链”。
2. **Solana 的低费高频** 更适合做轻量资产、战绩、排行榜和赛季进度。Solana 官方也专门维护了游戏示例、游戏 SDK 和游戏开发入口。
3. 黑客松评委通常会看：是否真的用到链、是否有用户价值、是否能继续扩展成产品。你这个题材天然能讲清楚三点。

#### 不要一开始就做“全链上战斗”

第一版最稳的架构是：

#### 链下

* 战斗循环
* 抽牌 / 出牌 / 敌人 AI
* 背包格子布局
* 绝大多数数值计算
* 动画、关卡、音效、存档草稿

#### 链上

* 玩家 Profile
* 赛季/周目结果摘要

* 排行榜提交

原因很现实：Solana 程序适合做 **状态确认、资产归属、可组合性**，但不适合把大量 roguelike 局内状态每一步都写链。Solana 官方开发资料也强调要关注计算消耗，并提供面向游戏的模板和 SDK。

#### 一套最适合黑客松的系统设计

#### 1. 最小可行玩法

* **背包系统**：6x4 或 5x5 格子

#### 游戏规则里最值得强调的创新点

你做成“背包英雄 + 卡牌 roguelike”，最好突出一个真正新鲜的核心机制：

#### 核心机制建议

**背包布局直接改写牌组行为**  
比如：

* 剑放在盾旁边：普通攻击获得格挡
* 法杖周围每有 1 个宝石：法术伤害 +1
* 弓放在背包边缘：首张远程牌免费
* 诅咒物品占格子，但可触发高风险高收益牌

这样你有三层策略：

1. 选什么物品
2. 怎么摆放
3. 战斗里怎么出牌

这会比“普通卡牌 + 普通 NFT”更容易打动评委。

### ChatGPT

结论先给你：

**黑客松版本，最优方案是：整场 run 结束后，或在少数“关键节点”与链上交互。**  
**不要做“每次操作/每轮抽牌都上链”，也通常不要做“每场小战斗都上链”。**

原因不是“不能”，而是 **产品体验、费用、失败率、开发复杂度** 都会明显变差。Solana 虽然手续费低，但每笔交易依然有基础费；而且即使交易失败，费用通常也会被收取。官方也明确强调，程序应尽量减少计算消耗，并注意 compute budget 与优先级费用。

* * *

#### 三种方案怎么选

#### 方案 A：每次操作 / 每轮抽牌都上链

比如：

* 抽 1 张牌发 1 笔交易
* 出 1 张牌发 1 笔交易
* 回合结束再发 1 笔交易

**我不建议。**

为什么：

1. **交互感会很差**  
    卡牌 roguelike 的节奏应该是连续、即时、顺滑。每步都签名或等待确认，会直接破坏可玩性。Solana 官方游戏开发指南也强调，链上交互更适合“小而关键”的集成；并专门提到游戏里可以通过自动批准交易来改善流畅度，侧面说明“频繁逐步签名”本来就是体验问题。
2. **费用会被放大**  
    Solana 每笔交易有基础费用，当前文档列的是 **每个签名 5,000 lamports**，还可能叠加优先级费用。高频小交易会把本来便宜的链，变成“总数不便宜”。
3. **失败面会变大**  
    任何一笔失败都可能打断局内流程，而且文档明确写了：**交易即使失败，费用也会被收取。**
4. **程序复杂度暴涨**  
    你得把完整战斗状态、背包状态、抽牌堆、弃牌堆、敌人状态全部做成链上状态机，还要持续控制 CU 和账户读写，黑客松里很容易把时间烧光。Solana 官方一直强调要优化 compute 使用。

**适合它的只有一种情况：**  
你做的是“全链上博弈”本身，链上可验证过程就是作品卖点。  
但你这个题材的卖点显然不是这个。

* * *

#### 方案 B：每场小战斗结束上链

比如：

* 一场战斗打完，提交 battle result
* 再进入下一场

**可以做，但不是最优默认。**

优点：

* 比逐步上链轻很多
* 每场战斗都能沉淀一些可验证结果
* 适合做“每日挑战赛”“闯关积分榜”

缺点：

* 还是会打断体验
* 会增加很多中间状态设计
* 一局 5 场战斗就至少 5 次链交互
* 你仍然要处理“玩家中途退出 / 断线 / 未提交 / 重开”之类问题

**什么时候适合：**  
你要把“每一场战斗”作为比赛单位，比如：

* 每日单关挑战
* Boss Rush 排行榜
* PvP 异步对战的单回合结算

对于你现在这个“背包英雄式 roguelike 卡牌游戏”首版，我觉得这不是最优起点。

* * *

#### 方案 C：整场游戏完成后一起上链

比如：

* run 开始时可选创建一个 run
* run 结束时一次性提交结果摘要

**这是我最推荐的默认方案。**

优点：

1. **最好玩**  
    局内所有操作都链下即时结算，游戏像正常独立游戏一样顺滑。
2. **最好做**  
    你只需要把链上程序做成：
    * `init_player`
    * `start_run`（可选）
    * `submit_run`
    * `unlock_achievement`
3. **最适合黑客松**  
    演示很清楚：  
    “游戏过程流畅 → 通关后战绩上链 → 排行榜 / 成就出现”
4. **最符合 Solana 在游戏里的自然位置**  
    官方游戏文档本来就把 Solana 放在游戏身份、资产、成就、支付、轻量游戏交互这些位置，而不是要求你把所有战斗逐 tick 写链。

* * *

#### 我建议你采用“1 + 1/2”结构

不是纯粹只在最后一次交互，而是：

#### 必要链上交互 1：进入 run 时

可选做一个轻量 `start_run`  
记录：

* player
* run_id
* seed_hash / ruleset_hash
* season_id
* start_ts

作用：

* 防止“先打完再伪造一个更好的开局”
* 给每日挑战 / 排行榜一个起点
* 给回放验证留钩子

#### 必要链上交互 2：run 结束时

做 `submit_run`  
记录：

* run_id
* final_score
* floor_reached
* win/lose
* deck_hash
* relic_hash
* replay_hash
* end_ts

#### 可选链上交互：关键奖励点

只在非常少的高价值事件上链，比如：

* 首次通关解锁成就
* 获得稀有 relic skin
* 赛季奖励 claim

这个结构比“只最后提交”更稳一点，但仍然不会破坏游玩体验。

* * *

### ChatGPT

会，而且不是“有点难”，而是：

**如果稀有卡牌掉落概率是在客户端本地算的，而你只在整局结束时把结果提交上链，那么玩家篡改掉落概率这类作弊，链本身通常看不出来。**

#### 先说本质

你现在讨论的是一种典型问题：

**“游戏过程在链下，结果在链上”**  
这类架构最大的安全弱点，不是交易伪造，而是 **结果生成过程不可见**。

如果以下事情都在客户端完成：

* 战斗奖励生成
* 掉落随机数
* 稀有度判定
* 奖励池抽样

那攻击者只要改前端 JS、改本地内存、挂代理、改 API 返回，最后再提交一个“看起来合法”的结果就行。  
链上只能验证：

* 这个玩家签了名
* 这个提交格式对不对
* 这个 run_id 没重复
* 这个账户关系合法不合法

但**很难验证“这个掉落是不是按正确概率算出来的”**。这也是为什么可验证随机数在链上游戏里会被单独当成一类基础设施来做。Switchboard 的文档就直接说明了：链上随机数很难，随机性通常需要由外部机制提供并把单一结果提交到链上，否则不同节点各自算随机会导致状态不一致。

#### 这类作弊为什么难检测

#### 1. 结果看起来可能“完全合理”

如果有人把稀有卡掉率从 1% 改到 5%，他不需要每局都出神卡。  
他只需要“略微提高”，长期看收益巨大，但单局数据看起来并不夸张。

这比“直接提交 99999 分”更难抓，因为后者很像异常值，前者更像运气好。

#### 2. 你最后只拿到摘要，不拿到过程

如果链上只存：

* 最终分数
* 通关层数
* 拿到的卡牌列表

你没有：

* 每次奖励节点的随机源
* 奖励池快照
* 抽样过程
* 每步战斗日志

那就几乎没法重演“这张稀有卡到底为何掉出来”。

#### 3. 即使有回放，也不等于能证明没作弊

如果回放日志也是客户端自己生成的，攻击者可以连日志一起伪造。  
所以“我有 replay 文件”不自动等于“我能证明结果真实”。

#### 三种可行防护方案

#### 方案 1：奖励随机性放到服务端，链上只接收服务端签名结果

这是**黑客松最实用**的。

流程：

1. 客户端打完一场战斗，向你的后端请求奖励结算
2. 后端根据：
    * run seed
    * 当前 floor
    * 玩家状态摘要
    * 战斗结果
    * 服务器私钥 / 密钥盐  
        生成奖励结果
3. 后端把结果和签名返回客户端
4. 客户端提交到链上
5. 链上程序验证“这是授权结算服务签过的结果”

#### 好处

* 简单
* 开发快
* 足够应付黑客松
* 能挡住绝大多数改前端概率的作弊

#### 坏处

* 你要信任自己的后端
* 这是“中心化公平”，不是“可公开验证公平”

#### 适合你现在吗

**非常适合。**

* * *

#### 方案 2：奖励随机性使用可验证随机数（VRF / commit-reveal）

这是**更正统的链上公平方案**。

Switchboard 的 Solana 随机数文档和教程就明确围绕 **verifiable randomness** 和 **commit-reveal** 模式展开，核心目标就是让结果不是客户端想改就能改。

流程可以是：

1. run 开始时，把 `seed_commit` 上链
2. 某个关键奖励点触发时，请求 VRF / 随机结果
3. 随机值返回后，由程序根据固定表计算掉落
4. 稀有奖励由程序或受程序约束的结算逻辑发放

#### 好处

* 公平性叙事非常强
* 非常适合向评委解释“链的价值”

#### 坏处

* 集成复杂度更高
* 对游戏节奏和工程时间更敏感
* 不适合把每个小奖励都这么做

#### 适合你现在吗

**适合只用在“高价值掉落”上，不适合全覆盖。**

比如：

* Boss 后的传奇卡掉落
* 赛季奖励箱
* 首通稀有 relic

普通白卡蓝卡就别上这个。

#### 方案 3：纯链下战斗，但赛后做“可验证回放”

这是一种折中：

1. 你固定整个 run 的初始 seed
2. 所有随机都由一个公开 PRNG 从 seed 推出
3. 客户端把完整动作序列记录下来
4. 后端或挑战者可以重放整局
5. 若重放结果与提交结果不一致，则判作弊

#### 好处

* 体验流畅
* 设计上很优雅
* 对“每日挑战 seed”很友好

#### 坏处

* 你必须做到**完全确定性模拟**
* 前后端、版本、浮点数、随机数调用顺序都要严格一致
* 仍然需要一个“谁来审计回放”的机制

#### 适合你现在吗

适合做第二阶段，不适合当首个安全底座。

1. **数学上能不能重放整局**：能
2. **能不能在 Solana 链上把整局完整重放一遍**：通常不该这么做

核心原因是，Solana 单笔交易的计算预算是有限的。官方文档目前写的是：默认每条非内置指令 200,000 CU，每笔交易上限 1,400,000 CU；而且优先费用按你请求的 CU 上限计，不是按实际使用量计。对一个包含多场战斗、很多奖励结算、路线选择、卡牌选择的 roguelike 来说，把整局完整 replay 塞进一次链上验证，基本会非常吃紧，甚至不现实。

#### 先说结论

**可行的实现方式不是“链上重跑整局”，而是“链下可重放，链上只做轻验证”。**

你可以把体系设计成：

* **seed 决定整局所有随机**
* **游戏逻辑完全确定性**
* **客户端保存动作序列**
* **链下任何人都能重放整局**
* **链上只验证少量承诺值和结果摘要**

这是能落地的。

* * *

#### 什么叫“能用 seed 验算整局”

只要你把整局游戏写成一个确定性状态机，就能验。

形式上类似这样：

```
state_0 = init(seed, ruleset, player_loadout)

state_1 = step(state_0, action_1)
state_2 = step(state_1, action_2)
...
state_n = step(state_{n-1}, action_n)
```

这里：

* `seed` 决定所有随机源
* `action_i` 是玩家第 i 步操作
* `step()` 是纯函数
* 同样的 `seed + 同样的 action 序列`，一定得到同样的最终结果

这样一局里就算有：

* 3 场战斗
* 8 次奖励选择
* 多次路线分支
* 每场不同的抽牌和掉落

只要这些都来自同一个确定性规则系统，就都能被重放验证。

* * *

#### 真正的关键，不是战斗多不多，而是这 4 点

#### 1. 所有随机都必须从 seed 派生

不能混入任何本地随机，比如：

* `Math.random()`
* 当前时间
* 帧率
* 设备差异
* 网络返回顺序

你应该做成：

```
rng_master = PRNG(seed)
battle_seed_1 = hash(seed, "battle", 1)
reward_seed_1 = hash(seed, "reward", 1)
shop_seed_1   = hash(seed, "shop", 1)
```

也就是不用“全局一直 next() 到底”，而是**按事件域分叉 seed**。

这样好处很大：

* 某个模块多调用一次随机，不会污染整局后续随机
* 更容易调试
* 更容易验证

* * *

#### 2. 游戏逻辑必须完全确定性

这条最重要。

你必须避免：

* 浮点数
* 不稳定排序
* 依赖对象遍历顺序
* JS 和 Rust 表现不一致的边界行为

最好统一成：

* 全部整数数值
* 明确固定排序规则
* 每个事件处理顺序固定
* 随机调用点固定

否则你会遇到一个很烦的问题：

**不是玩家作弊，而是你自己 replay 不一致。**

* * *

#### 3. 必须记录“动作序列”，不是只记录最终结果

只提交：

* 最终分数
* 最终卡组
* 最终 relic

这是不够的。

因为同一个 seed 下，玩家中间不同选择会走向不同结局。  
所以你必须至少能拿到：

* 路线选择
* 奖励三选一选择
* 商店购买选择
* 战斗中的出牌/结束回合/目标选择
* 丢弃/使用消耗品等关键动作

最简单的做法是把一局编码成一串动作日志：

```
[
  choose_path(2),
  reward_pick(1),
  play_card(card_7, target_0),
  end_turn,
  play_card(card_3, target_1),
  ...
]
```

有了 `seed + action log`，链下就能完整重放。

* * *

#### 4. 规则版本必须固定

否则你会遇到另一个问题：

* 玩家玩的时候是 v0.1.2
* 你验证的时候程序已经更新到 v0.1.3

那 replay 可能直接变掉。

所以每个 run 都要绑定：

* `ruleset_version`
* `content_hash`
* `drop_table_hash`

链上至少要记住这些摘要值之一。

* * *

#### 为什么不建议链上完整验整局

因为你的一局可能包含上百步状态转移。

而 Solana 的交易费用和优先费用又和你请求的计算预算相关；官方明确写了优先费用取决于你请求的 CU limit，不是实际用了多少。也就是说，就算你勉强把 replay 塞进链上，成本、失败率和开发复杂度都不会好看。

更实际的问题是：

* 一次 replay 可能循环很多回合
* 每回合要处理多张卡牌、状态效果、掉落逻辑
* 多场战斗叠加后，CU 很容易爆
* 一旦验证失败，交易还是要付费；Solana 官方费用文档明确写了，交易失败也会收取费用。

所以架构上应该避免“链上重放整局”。

* * *

#### 真正可落地的验证架构

#### 方案 A：链下完整 replay，链上只验摘要

这是我最推荐的。

#### run 开始时上链

提交一个 `RunCommit`：

* player
* run_id
* seed_commit 或 daily_seed_id
* ruleset_hash
* deck_init_hash

#### 游戏过程中链下运行

本地保存：

* action log
* 中间状态哈希链

#### run 结束时上链

提交：

* run_id
* final_score
* final_state_hash
* action_log_hash
* optional replay_root

#### 链上只验证

* 这个 run 是否已注册
* seed 是否匹配 commit
* ruleset_hash 是否一致
* final submission 是否格式合法
* 是否重复提交

#### 真正的“验整局”

由链下 verifier 做：

* 客户端本地
* 社区旁观者
* 比赛裁判脚本
* 任何人下载动作日志后重放

这叫“公开可验证”，但不是“链上逐步执行”。

* * *

#### 方案 B：分段验证，而不是整局验证

如果你很想让链更强参与，可以把一局拆段：

* 地图段
* 每场战斗段
* 每次奖励段

每一段结束产出一个 `segment_hash`：

```
segment_1 = hash(seed_1, actions_1, state_end_1)
segment_2 = hash(seed_2, actions_2, state_end_2, segment_1)
...
```

最后链上只收：

* 第一段承诺
* 最终段摘要
* 或少量 checkpoint

这样比“整局一次性验”更稳，也更容易抽查。

* * *

#### 你问的“有可能验算吗”，答案是：有，但要换成“可重放证明”

你不该把目标写成：

> 在 Solana program 里重跑整局游戏

你该把目标写成：

> 给每一局生成一个任何人都能链下重放的证明包，链上只保存承诺和结果摘要

* * *

#### 一个适合你的具体设计

#### 1. seed 结构

```
master_seed
map_seed      = H(master_seed, "map")
battle_seed_i = H(master_seed, "battle", i)
reward_seed_i = H(master_seed, "reward", i)
shop_seed_i   = H(master_seed, "shop", i)
```

#### 2. 一局日志内容

```
RunProof {
  run_id,
  ruleset_hash,
  master_seed_reveal,   // 或 daily seed 引用
  initial_loadout_hash,
  actions[],
  checkpoints[],
  final_state_hash,
  final_score
}
```

#### 3. checkpoint

每场战斗结束记一个：

```
Checkpoint {
  index,
  phase_type,           // battle / reward / shop / route
  start_state_hash,
  actions_hash,
  end_state_hash
}
```

#### 4. 链上存什么

只存：

* player profile
* run commit
* run result summary
* final_state_hash
* action_log_hash
* maybe checkpoint_root

#### 5. 谁来验

* 前端本地可一键 replay
* 开源 verifier 脚本可重放
* 排行榜前 N 名强制提交完整 proof 包供任何人复核

这就足够去中心化了。

* * *

#### 这个方案的缺陷

#### 缺陷 1：链上不是“自动判真伪”

链本身不会帮你把整局算完。  
所以严格说，这是：

**链上承诺 + 链下可验证**

不是“纯链上执行”。

#### 缺陷 2：proof 包会变大

一整局动作很多，日志体积会上去。  
解决方式通常是：

* 链上只存 hash
* 详细日志放本地或去中心化存储
* 只对高排名 run 要求公开完整日志

#### 缺陷 3：你必须把规则写得非常规整

也就是前面说的确定性要求非常高。  
这其实是最大工程挑战。

* * *

#### 对你这游戏最现实的建议

#### 黑客松版

做这套：

* **每日挑战模式**
    * 每天一个公开 seed
    * 所有人同 seed
    * 提交 final result + action_log_hash
    * 前几名公开完整 replay
* **普通 run 模式**
    * 不做强验证
    * 只做轻量上链记录

这样你同时拥有：

* 一个真正可验证、可讲故事的模式
* 一个更自由、好玩的普通模式

这比试图“所有模式都链上严验”现实得多。

* * *

#### 最后一句话

**一局里包含数场战斗、数次选择、很多卡牌决策，并不妨碍验算；真正决定可不可验的，是你能不能把整局写成“seed 驱动 + 动作驱动 + 完全确定性”的状态机。**  
但在 Solana 上，最合理的实现不是“链上重放整局”，而是“链下重放整局，链上只验证承诺和摘要”。这也是更符合 Solana 计算预算与费用模型的做法。

### fffg

* * *

### ChatGPT

你抓到的是关键点：**如果协议一开始没有把“申诉/挑战/回滚排行榜”的流程写进链上规则里，事后发现作弊，链本身不会自动改榜。** 我先确认一下 Solana 官方关于程序、账户状态和交易执行的最新表述，再给你一个真正能落地的“可挑战排行榜”设计。

* * *

### ChatGPT

对，你理解得很准：

**如果链上只验证“seed/格式/摘要是否自洽”，那它只能防“明显不合法的提交”，防不了“过程作假但摘要看起来合法”的提交。**  
而且更关键的是，**一旦协议里没有预先设计“挑战—裁决—更新排行榜”的链上流程，事后就不能靠一句“他作弊了”去改共识。** 在 Solana 里，程序本身无状态，所有可变状态都在数据账户里；账户数据只能由其 owner program 通过指令改写；交易又是原子执行的，所以排行榜是否能被修正，取决于你有没有在程序里提前写好可执行的修正规则。

所以答案不是“链下验出作弊后怎么广播到链上、让链自动改榜”，而是：

#### 你要把排行榜设计成“可挑战排行榜”，不是“立即终局排行榜”

最实用的做法是把排行榜分成两层：

#### 第一层：临时榜

玩家提交 run 后，先进入 **pending / provisional leaderboard**。  
这一步链上只检查：

* run 是否注册过
* seed / commit 是否匹配
* 提交格式是否正确
* 基本摘要是否自洽

这类检查适合链上做，因为只涉及少量账户和少量字段。Solana 交易和指令本来就是按“明确传入账户 + 明确指令数据”的方式执行的，不会替你自动读取一整局外部过程。

#### 第二层：终局榜

给每个 run 一个 **challenge window**，比如 1 小时、24 小时，或者“到每日赛结束前”。  
在这个窗口内，任何人都可以对该 run 发起 challenge，提交：

* run 的完整动作日志或其可取回引用
* challenge 类型
* 作弊证明材料
* 必要押金

如果窗口结束且无人成功挑战，该条目才从 provisional 进入 final leaderboard。  
这不是“修改共识”，而是**排行榜账户本来就处于“待终局”的协议状态**，后续通过程序指令合法推进到 final。Solana 的状态就是账户数据，程序只要被调用、并且拿到了可写账户，就可以按你定义的规则更新这些状态。

* * *

#### 也就是说：不是“广播证明后让链改历史”，而是“在历史尚未终局前触发协议内的争议流程”

这是两个完全不同的东西。

你不能指望：

* 链下有人跑了 verifier
* 在 X 上发帖说某人作弊
* 链自己就把旧交易推翻

因为 Solana 交易一旦执行成功，它本身不会被应用层随意撤销；应用层能做的是**再发一笔新的合法交易**，去更新你的排行榜账户、把某条成绩标记为 invalid、扣罚保证金、重排名次。交易是原子执行的，状态回滚只发生在当前交易失败时，不会因为后面“有人不同意”而自动回滚旧的应用状态。

* * *

#### 你真正需要的，是一套“乐观提交 + 挑战期 + 裁决”的协议

我建议你直接做成下面这个结构。

#### 方案：Optimistic Leaderboard

#### 1. `submit_run()`

玩家提交：

* `run_id`
* `seed_commit` 或 `daily_seed_id`
* `result_summary`
* `score`
* `final_state_hash`
* `action_log_hash`
* `proof_uri` 或 `proof_root`
* 押金 `bond`

链上做轻验证后：

* 生成 `RunRecord { status = Pending }`
* 把它挂到 provisional 榜单
* 记录 `challenge_deadline`

#### 2. `challenge_run()`

挑战者在时限内提交：

* `run_id`
* `challenge_type`
* `evidence_hash` / `proof payload`
* 挑战押金

链上把 run 状态改为：

* `UnderChallenge`

#### 3. `resolve_challenge()`

这里有两条路：

#### 路 A：链上可裁决的挑战

只处理那些**链上能低成本验证**的作弊，例如：

* seed reveal 不匹配 commit
* 超出允许卡池
* 最终 relic 数量超过规则上限
* 不可能分数
* checkpoint hash 链断裂
* 明显违反状态转移边界

这类可以由程序直接判：

* challenge 成功 → run 记为 `Invalid`
* 挑战者拿回押金并获奖励
* 提交者被罚 bond
* 榜单重排

#### 路 B：链下可验证、链上可执行裁决

更复杂的作弊，如“中间某回合偷偷改了掉率”，通常不适合链上完整重放。  
这时你需要协议里预留一个 **裁决入口**，例如：

* 多签裁判团
* 委员会签名
* 后续你如果真做 zk/证明系统，也可以接证明验证

他们链下跑 verifier，确认作弊后，再调用：

* `slash_run(run_id, signed_verdict)`
* 或 `resolve_challenge(run_id, verdict)`

这样“广播到链上”的不是舆论，而是**符合你协议格式的裁决指令**。

* * *

#### 这就回答了你最担心的问题

你担心的是：

> 会不会榜上先挂满作弊者，虽然链下知道他们作弊，但链上没法改？

答案是：

**如果你做“立即终局榜”，会。**  
**如果你做“可挑战的乐观榜”，就不会成为致命问题。**

因为在后者里：

* 新成绩先只是 `Pending`
* 只有过了挑战期才 `Final`
* 被挑战成功的条目不会进入终局榜
* 即便已经进入终局榜，只要协议允许，也可以通过专门的惩罚/失效指令把其标记为 `Slashed/Invalidated`

* * *

#### 但这里有一个更现实的判断

#### 如果你不想中心化，又不想做链上完整 replay

那你必须接受一个现实：

**最终裁决一定会存在“谁有权认定作弊”的问题。**

因为仅靠“seed + 最终摘要”，不足以让链自动判断所有过程型作弊。  
你至少要在三种取舍里选一种：

#### 选项 A：只支持链上可判的作弊类型

优点：

* 最去中心化
* 最干净

缺点：

* 能抓到的作弊有限
* 很多“过程型作弊”抓不到

#### 选项 B：引入挑战委员会 / 仲裁多签

优点：

* 实际可用
* 能处理复杂作弊

缺点：

* 有一定中心化

#### 选项 C：上更强的证明系统

比如未来做：

* zk proof
* 或极强的可验证回放证明

优点：

* 最理想

缺点：

* 黑客松几乎不现实

* * *

#### 对你项目最合理的落地方式

我建议你按模式分层。

#### 普通模式

不做强共识排行榜，只做：

* 上链成绩展示
* 社区榜
* 不带高价值奖励

这样哪怕有人作弊，也只是污染展示层，不影响资产安全。

#### 每日挑战模式

这里做你真正的“严肃排行榜”：

* 固定 daily seed
* 强制提交 `action_log_hash`
* 前 N 名必须公开完整 replay
* 榜单先 Pending
* 挑战期后 Final
* 仅 Final 榜单发奖励

#### 高价值奖励

只从 Final 榜单发：

* NFT / 成就
* 赛季奖励
* 稀有 relic skin

这样你就把最难的问题，集中在少量高价值 run 上处理，而不是要求每一局都做到链上完美裁决。

#### 我给你一个很实用的链上状态设计

#### `RunRecord`

```rust
status: Pending | UnderChallenge | Final | Invalid
submitter: Pubkey
score: u64
seed_commit: [u8; 32]
action_log_hash: [u8; 32]
final_state_hash: [u8; 32]
proof_root_or_uri_hash: [u8; 32]
challenge_deadline: i64
bond: u64
```

#### `ChallengeRecord`

```rust
run_id: u64
challenger: Pubkey
reason_code: u8
evidence_hash: [u8; 32]
status: Open | ResolvedSuccess | ResolvedFail
bond: u64
```

#### `LeaderboardEntry`

不要单独把名次写死。  
更好的办法是：

* 榜单读取时只统计 `Final` 状态的 run
* `Pending/Invalid` 不参与最终奖励结算

这样你甚至不需要频繁“重排链上榜单数据结构”，前端或索引器根据状态计算最终榜单即可。

#### 你说的“修改共识”这个词，我帮你纠正一下

从应用设计上看，不是“修改共识”，而是：

**通过新交易更新应用状态。**

* `proof_uri` 或 `replay_uri`
* `run_id`
* `minted_at`
* `verified_level`

这样 NFT 会变成一个 **可验证的挑战成就凭证**，而不是一张只有图片的卡。  
Solana 现在常见的 NFT 元数据方式就是：链上 metadata 账户保存基础字段和一个 `uri`，再由这个 `uri` 指向链下 JSON；官方文档也明确说明了链上 metadata 通常只放名称、符号、URI 等，而扩展信息放在 URI 指向的 JSON 中。

#### 但不要误以为“写进去就自动防作弊”

如果作弊者也能 mint 出 NFT，那么：

* 他同样可以把假的 `score`、假的 `action_log_hash`、假的 `proof_uri` 写进去
* NFT 仍然会存在于链上
* 钱包仍然可能显示它
* 市场也不一定自动把它当成“无效”

所以真正的问题不是“能不能写进 metadata”，而是：

> **谁有权 mint，mint 前验证到什么程度，mint 后如何标记合法性。**

* * *

#### 为什么这条路有价值

#### 价值 1：让 NFT 从“图片收藏品”变成“可审计成就”

这很适合你的黑客松定位。

你可以把 NFT 定义成：

**Daily Challenge Achievement NFT**

它的元数据不是只有：

* 名字
* 图片
* 稀有度

还包含：

* 哪一天的 daily seed
* 哪个版本规则
* 玩家得分
* replay 证明地址
* 是否通过本地验证

这样别人拿到这张 NFT，不只是“看图”，而是可以追溯它的来源。

* * *

#### 价值 2：你可以把“验证状态”也显式化

比起只写原始数据，更好的做法是加一个字段：

```json
{
  "attributes": [
    {"trait_type": "Daily Seed", "value": "2026-04-23"},
    {"trait_type": "Score", "value": 2480},
    {"trait_type": "Tier", "value": "Gold"},
    {"trait_type": "Ruleset Hash", "value": "0x..."},
    {"trait_type": "Action Log Hash", "value": "0x..."},
    {"trait_type": "Final State Hash", "value": "0x..."},
    {"trait_type": "Verification Status", "value": "Self-Reported"}
  ]
}
```

然后以后如果你有更强的验证器，可以把状态升级成：

* `Self-Reported`
* `Replay-Published`
* `Community-Verified`
* `Officially-Verified`
* `Disputed`

这会非常有用。

* * *

#### 但这条路的硬限制是什么

#### 限制 1：NFT 元数据本身不是裁判

Solana / Metaplex 的常见元数据结构里，链上存的是：

* name
* symbol
* uri
* creators
* mutability 等

详细的扩展属性通常在 `uri` 指向的链下 JSON 里。Solana 官方文档还特别建议把 JSON 存在可靠的持久存储上，比如 Arweave、IPFS 或稳定 CDN。

这意味着：

* 元数据是“描述”
* 不是“自动执行验证规则”的地方

也就是说，metadata 可以帮助别人验证，但不会替你执行惩罚。

* * *

#### 限制 2：如果元数据可更新，作弊者也可能改；如果不可更新，你也难以后补标签

官方文档说明 metadata 是否可变，取决于 `is_mutable` / update authority 设计；如果账户可变，update authority 可以更新 metadata。

所以你会遇到一个取舍：

#### 可变 metadata

优点：

* 你以后能把 `Verification Status` 从 `Pending` 改成 `Verified/Disputed`

缺点：

* 需要一个 update authority
* 这引入一定中心化
* 用户会问：是不是官方能随便改 NFT

#### 不可变 metadata

优点：

* 更“硬”
* 收藏品叙事更强

缺点：

* 一旦 mint 出去，事后很难补上“此 NFT 有争议”的官方状态

* * *

#### 限制 3：市场价值不是由“写了 proof”自动产生的

你说“作弊者的 NFT 会轻易验证不合法，失去价值”，这句话只在一个前提下成立：

> **你的前端、社区和可能的交易场景，都认可某种验证标准。**

否则链上只会看到：

* 一个真实存在的 mint
* 一个真实存在的 metadata
* 一个也许写了假 proof 的 URI

链不会替你说“这张作废”。

所以更准确的说法是：

**proof 写进 NFT metadata 可以让“合法 NFT 更容易被识别”，但不能保证“非法 NFT 自动失去价值”。**

* * *

#### 真正合理的做法：把“是否有效”设计成可读状态，而不是寄希望于市场自己理解

我建议你做成两层。

#### 第一层：NFT 记录 proof

在 metadata JSON 里放：

* `run_id`
* `day_id`
* `seed_commit`
* `ruleset_hash`
* `score`
* `action_log_hash`
* `final_state_hash`
* `proof_uri`

#### 第二层：再加一个“官方可查询的验证状态”

这个状态不要只靠 NFT 自己说自己合法。  
更稳的是由你的 game program 或一个独立 registry 账户维护：

* `mint -> Pending`
* `mint -> Verified`
* `mint -> Disputed`
* `mint -> RevokedForRewards`（如果你以后需要）

这样：

* NFT 本体仍然存在
* 但你的官网、排行榜页、游戏内卡册，只承认 `Verified`

这就足够了。

* * *

#### 对黑客松来说，最适合你的实现

#### 方案 A：NFT 元数据里写 proof，NFT 默认是“纪念品”

这是最稳的。

#### mint 条件

* 每日挑战完成
* 达到某个分数门槛

#### metadata 里写

* `daily_seed_id`
* `score`
* `tier`
* `action_log_hash`
* `proof_uri`
* `verification_status = "Unverified"` 或 `"Replay-Available"`

#### 前端展示

* 你的官网提供一个 “Verify this NFT” 按钮
* 用户点一下，本地下载 replay / action log
* 重放成功后前端显示 “Replay matched”

这样就很好了。

#### 这时候即便有作弊者

* 他能 mint 出某些 NFT
* 但你的网站可以不把它显示为 verified achievement
* 社区也能区分

对于黑客松，这已经足够强。

* * *

#### 方案 B：NFT 先 mint，后补验证状态

如果你想再前进一步，可以保留 update authority，后续把 metadata 里的状态改成：

* `Pending`
* `Verified`
* `Disputed`

技术上是可行的，因为可变 metadata 可以被更新。

但我不建议你一开始就把这个做太重。  
黑客松里更简单的做法是：

* metadata 固定
* 另做一个链上或链下 registry 记录验证状态

* * *

#### 我建议你不要把完整 action 全塞进链上 metadata

这是一个实现细节，但很重要。

#### 不要做

* 把整段 action log 直接写进链上 metadata 字段

原因：

* 标准 metadata 结构本来就不是给大体量游戏回放数据设计的
* 官方文档强调常规做法是链上 metadata 存 URI，详细 JSON 放链外。

#### 应该做

链上或 metadata JSON 里只存：

* `action_log_hash`
* `proof_uri`

然后：

* `proof_uri` 指向 IPFS/Arweave 上的 replay 文件
* 重放器下载文件验证哈希是否一致

这才是正常结构。

* * *

#### 你这个思路最好的表达方式

不是：

> 把数据写进 NFT，作弊者 NFT 就会自动失去价值

而是：

> 把每日挑战证明嵌入 NFT 元数据与可追溯 proof 中，让合法成绩具备公开可审计性；官方前端和社区工具可据此区分已验证与未验证成就 NFT。

这个表述更准确，也更专业。

这样非常合理，而且黑客松内可实现。
